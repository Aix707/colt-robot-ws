# AGENTS.md

适用于 `/home/xia/桌面/catkin_ws` 整个仓库。若子目录有更近的 `AGENTS.md`，以更近规则为准。

## 目标

- 代码要精简、可读、易维护。
- 优先减少复杂度，不为“完整性”保留无用层、无用状态机、无用 demo。
- 以当前代码和实际可运行结果为准，不编造不存在的文件、节点、命令、话题或测试结果。

## Python 规则

- Python 优先使用成熟第三方库，尽量少自己写通用功能。
- 能用现成库解决的功能，不要重复实现。
- 能复用仓库里已有公共函数、脚本、消息、配置的，不要再复制一份。
- 自己写的代码应尽量只保留“业务相关的薄封装”，不要手写大段底层通用逻辑。
- 如果缺少依赖，先明确需要安装什么库，不要退回去造轮子。

## 实现风格

- 功能要收敛：一个模块只做自己的核心职责。
- 允许为同一目标同时修改多个相关文件，但改动要朝着更简单的结构收敛。
- 优先删重复、合并职责、统一入口、减少中间层。
- 不要加入与当前目标无关的安全性设计、抽象层、兼容层或预留架构。
- 注释只保留必要说明，避免注释比代码还复杂。

## 当前边界

- 本仓库是 ROS Noetic / catkin 工作空间，常规构建命令是 `catkin_make`。
- 当前新开发主要在 `src/colt/`。
- 外部训练项目在 `/home/xia/桌面/colt_trainer_py`，不属于本仓库提交范围。
- `src/colt/DEVELOPMENT_FLOW.md` 只是路线参考，不要求每次机械按阶段推进。

## 必须保留的边界

- 未经用户明确要求，不发布 `/cmd_vel`、`/wpv4_pt/joint_ctrl_degree`、机械臂控制话题或抓取 action。
- 不让感知节点直接驱动真实底盘、云台或机械臂。
- 涉及真实运动、硬件使能或公开接口变更时，先确认输入、输出、坐标系和停止条件。
- 坐标输出必须带 `header.stamp` 和 `header.frame_id`，并说明坐标来源或约束。

## 修改边界

- 对第三方或旧功能包保持克制，例如 `iai_kinect2`、`wpv4_bringup`、`wpv4_behaviors`、`wpv4_tutorials`、`wpm2_*`。
- 只有当前任务确实需要时，才修改这些旧包。
- 不提交 `build/`、`devel/`、数据集、模型权重、rosbag、点云、大型中间产物和临时日志。
- 不把 `/home/xia/桌面/colt_trainer_py` 的内容混进本仓库，除非用户明确要求迁移。

## 最小工作习惯

- 开始前看 `git status --short --branch`。
- 先搜索再改，优先用 `rg` / `rg --files`。
- 改完至少检查 `git diff --check`。
- 只改文档时，可以不跑 `catkin_make`，但要说明。
- 改了 ROS 消息、`CMakeLists.txt`、`package.xml`、launch 或运行链路时，尽量跑对应验证。
- 新增或修改 Python 脚本时，至少跑 `python3 -m py_compile <script.py>`。
